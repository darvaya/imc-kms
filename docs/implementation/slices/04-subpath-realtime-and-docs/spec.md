# Slice: subpath-realtime-and-docs

## Summary
Final slice. Prefix the WebSocket upgrade-handler paths in `server/services/websockets.ts` (`/realtime`) and `server/services/collaboration.ts` (`/collaboration`) with `BASE_PATH`, prefix the matching client paths in `app/components/WebsocketProvider.tsx`, and update `docs/apache-vhost.conf` and `.env.example` to document the shared-subdomain layout as the canonical deployment. After this slice lands, real-time presence and Hocuspocus collaborative editing work end-to-end under `/kms/`, and ops has a single documented deployment recipe (with WebSocket reverse-proxy config) for the new layout.

## Motivation
WebSocket upgrade handlers register on the raw `http.Server` (not on Koa) — koa-mount cannot rewrite their paths, so they're the only server-side surface that needs explicit `BASE_PATH +` prefixing. With slices 1–3 in place, every other piece of the app works under `/kms/`, but opening a document still throws connection errors in the console: the Socket.IO presence connection (`/realtime`) and the Hocuspocus collaborative-editing connection (`/collaboration/<docId>`) both upgrade against root paths that no longer exist behind the reverse proxy. Bundling the doc updates with this slice means ops gets the new layout, the WebSocket rewrite rule, and the env-example update as a single coherent change — no risk of merging code that "works" but ships with stale Apache config.

## User stories
- **As an end user collaborating on a document**, I want to see my colleague's presence avatar and live cursor under `/kms/`, and have my edits sync to them in real time, so the sub-path deployment is fully feature-equivalent to the dedicated-subdomain deployment.
- **As a devops engineer onboarding a new ops teammate**, I want `docs/apache-vhost.conf` and `.env.example` to show the shared-subdomain layout (with the WebSocket rewrite rule) as the canonical deployment recipe, so deploying a new sub-path takes one config copy + one env tweak instead of reverse-engineering the code.

## Technical notes
- **Why WebSocket handlers are outside the Koa mount**: both `websockets.ts` and `collaboration.ts` register `'upgrade'` event handlers directly on the raw `http.Server` returned by `http.createServer()` in `server/index.ts`. The HTTP-upgrade handshake happens before Koa's middleware pipeline runs, so koa-mount has no opportunity to rewrite `req.url`. This is also why `cors.origin` and the Origin-check on line 70 read directly from raw request headers, not from `ctx.headers`. Explicit prefixing is the only viable approach.
- **`COLLABORATION_URL` already propagates the path**: `server/env.ts:241-243` derives `COLLABORATION_URL` from `URL` by stripping trailing slash and replacing `http` with `ws`. With `URL=http://host/kms`, `COLLABORATION_URL` becomes `ws://host/kms`, and the client's `${env.COLLABORATION_URL}/collaboration` becomes `ws://host/kms/collaboration` — which matches what the server will now listen for. No client-side change is required for the Hocuspocus connection. Add an assertion in the integration test that this round-trips correctly.
- **Microsoft Entra deploy prerequisite (out of scope but critical)**: the Microsoft Entra app registration's redirect URI must be updated to `https://appstpcid.imcpelilog.co.id/kms/api/better-auth/callback/microsoft` *before* the deploy switch-over. If the redirect URI in Azure doesn't match what better-auth sends, OIDC login fails with `AADSTS50011: redirect URI mismatch`. Call this out in the PR description and in the updated `.env.example` deployment-notes block.
- **Apache WebSocket rewrite scoping**: the WebSocket rewrite rule must be scoped *to the sub-path*, not host-wide. With multiple apps on `appstpcid`, a host-wide rule (`RewriteRule /(.*) ws://127.0.0.1:3100/$1`) would intercept WebSocket upgrades destined for `inventory journal plog` or `pgl` and route them to KMS. The correct rule is `RewriteRule /kms/(.*) ws://127.0.0.1:PORT/kms/$1 [P,L]` so only `/kms/...` upgrades route to the KMS process.
- **Apache `ProxyPass` trailing slash**: use `ProxyPass /kms http://127.0.0.1:PORT/kms` (no trailing slash on either side) so the prefix is preserved in the forwarded request. Adding a trailing slash on either side strips/adds the slash and can break path matching at the Koa mount.
- **Origin check robustness**: `server/services/websockets.ts:70` uses `env.URL.startsWith(req.headers.origin)`. With `env.URL=http://host:PORT/kms` and `req.headers.origin=http://host:PORT`, the comparison `"http://host:PORT/kms".startsWith("http://host:PORT")` returns `true` — works as intended. Verify in the integration test.
- **Memory update (post-deploy)**: once the deploy is verified live, the user-level memory at `.claude/projects/.../memory/project_deployment.md` should drop the "no subpath support" claim and document the new layout. This is operationally a separate step (not a code change), so it's intentionally not in this slice's acceptance criteria.
