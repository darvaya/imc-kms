## Key decisions

- **Helper extraction over jest mock for WebsocketProvider.** Plan §8 risk #2 left the choice between (a) extracting a `getRealtimePath()` helper and (b) mocking `socket.io-client`'s `io` export. Initial implementation used (a) inside `WebsocketProvider.tsx` itself, but the test still failed to load because importing the helper pulled the `@observer`-decorated class along with it (review §3.4). Resolution: moved `getRealtimePath` into `app/components/getRealtimePath.ts` (its own module). The component now imports from there, and the test imports from there too — `WebsocketProvider.tsx` never loads in the test path. Cost: one extra file. Benefit: 4 client tests now run cleanly without touching the `app/` jest setup.
- **Realtime fallback path duplication accepted.** Plan §8 risk #1 weighed (a) a one-line comment on `collaboration.ts:113` vs (b) a shared `realtimePath()` constant in a new file. Chose (a) — minimal-diff principle. The comment explicitly directs future contributors to `websockets.ts:33`. If `/realtime` ever migrates to a different segment (e.g., `/socket.io`), both sites need updating; the comment is the trip-wire.
- **`COLLABORATION_URL` round-trip asserted via derivation expression, not field access.** Review §3.3 caught that `env.COLLABORATION_URL` is a one-shot class-field initializer (server/env.ts:252-254), not a getter — mutating `env.URL` in tests does not recompute it. Rather than convert `COLLABORATION_URL` to a getter (would expand slice scope) or fork a fresh `Environment` instance per test (heavyweight), the tests assert the derivation expression directly: `"https://app.outline.dev/kms".replace(/\/$/, "").replace(/^http/, "ws")`. Same logic, lower test fidelity, zero scope creep. Real round-trip evidence comes from manual smoke step 18 (DevTools shows `ws://localhost:3100/kms/collaboration/<docId>`).
- **Path-less origin tests use a non-cloud-hosted URL fixture.** `setup.ts:46` sets `env.URL = "https://app.outline.dev"`, which is in the `isCloudHosted` allowlist (server/env.ts:835-841). Cloud-hosted mode short-circuits the on-prem origin check. The path-less describe block in `websockets.test.ts` overrides to `https://kms.example.test` (non-cloud, no path → `BASE_PATH=""`) so the origin assertions actually exercise the code under test.
- **`docs/apache-vhost.conf` ALTERNATIVE block is commented-out, not removed.** Existing dedicated-subdomain deployments (and any reader looking up "how do I deploy without sub-path?") still find a working reference. The PRIMARY shared-subdomain block is uncommented and ready to use; the dedicated-subdomain block is documented as a comment.
- **Manual smoke walkthrough was completed live**, unlike slices 02 and 03 (which deferred). Both Milestone A (path-less regression) and Milestone B (sub-path acceptance) plus Milestone C (cleanup) all passed end-to-end with two browsers, OIDC sign-in for two distinct users, presence avatars, live-cursor sync, disconnect timing, and DevTools WS panel verification.

## Known issues / tech debt

- **Microsoft Entra redirect URI is a deploy prerequisite, not a code change.** When deploying to a sub-path, the Microsoft Entra app registration's redirect URI in Azure must be updated to `https://appstpcid.imcpelilog.co.id/kms/api/better-auth/callback/microsoft` *before* the deploy switch-over. Skipping this fails OIDC with `AADSTS50011: redirect URI mismatch`. Documented in `.env.example` lines 73-77 and in the DEPLOYMENT NOTES block, but this is *operational* — not enforceable by code. Whoever performs the deploy must read the env file.
- **Realtime path string is duplicated across two files.** `server/services/websockets.ts:33` constructs `${env.BASE_PATH}/realtime` and `server/services/collaboration.ts:113` re-derives it for the cross-service fallback. A 2-line comment on `collaboration.ts:110-111` warns future contributors to keep them in sync. A cleaner refactor would export a shared `realtimePath()` constant from a new `server/services/realtimePath.ts`. Out of scope for this slice (plan §8 risk #1); recommend a small follow-up cleanup PR if the value is ever referenced again.
- **`env.COLLABORATION_URL` is a one-shot field initializer.** The field is computed once at `Environment` construction time. In production, `env.URL` doesn't mutate post-startup, so this is fine. In tests, mutating `env.URL` does NOT recompute `env.COLLABORATION_URL`. If a future test needs to assert against `env.COLLABORATION_URL` for a sub-path layout, either (a) replicate the derivation expression directly, or (b) convert `COLLABORATION_URL` to a getter (`get COLLABORATION_URL() { ... }`) — same shape as `BASE_PATH`. Out of scope for this slice.
- **Apache `apache2ctl configtest` not run.** Plan §7 step 19 marked it optional ("config-only, no live deploy"). Skipped in favor of visual inspection (the rewrite/proxy lines all reference `/kms/` and `127.0.0.1:3100` consistently; multi-tenant safety comment is in place; trailing-slash invariants on `ProxyPass` are preserved). Recommend running `apache2ctl configtest` against the deployed copy on the target host before the deploy switch-over.
- **Memory file `.claude/projects/.../memory/project_deployment.md` not updated.** Per spec §scope/out-of-scope, the memory should drop the "no subpath support" claim post-deploy. Operationally separate from this PR; flagged for the user to update once the deploy is verified live.
- **Slice 02's startup-drift warning still not implemented.** Slice 02's `completed.md` "Dependencies for future slices" suggested adding a startup warning in `server/index.ts` that compares `env.BASE_PATH` against the path baked into Vite's `manifest.json`. Out of scope per spec §scope (this slice's scope listed only the WebSocket files + Apache + .env.example + tests). Recommend a separate `chore: warn on BASE_PATH/manifest drift` PR.
- **`reflect-metadata` polyfill gap in `app/` jest project remains.** Slice 03's known issue. The helper-extraction in this slice sidesteps the gap for the new test, but the broader gap (e.g., `app/models/User.test.ts`, `app/models/Collection.test.ts` still failing on `Reflect.metadata`) is unchanged. Recommend a separate `chore: import reflect-metadata in app jest setup` PR (slice 03's `completed.md` proposed the same).

## Dependencies for future slices

There are no further sub-path slices planned — this is the final slice in the series. After this lands and is deployed, the entire app works end-to-end under `${env.BASE_PATH}` (e.g., `/kms`).

For future contributors building on this work:

- **`getRealtimePath()` (`app/components/getRealtimePath.ts`)** is the canonical client-side accessor for the Socket.IO upgrade path. New code that opens a Socket.IO connection should call this helper, not hard-code `/realtime`.
- **`${env.BASE_PATH}/realtime` and `${env.BASE_PATH}/collaboration`** are the canonical server-side WebSocket upgrade paths. New WebSocket upgrade handlers (if any) should follow the same pattern: derive the `path` constant from `env.BASE_PATH` at `init()` time, then `req.url?.startsWith(path)` for routing decisions.
- **WebSocket reverse-proxy scoping rule** is documented in `docs/apache-vhost.conf` and applies to any future WebSocket service deployed on a multi-tenant hostname: scope the `RewriteRule` to the app's prefix, never host-wide. A host-wide rule would intercept WebSocket upgrades for every co-tenant app on the hostname.
- **`COLLABORATION_URL` derivation is path-preserving.** When introducing new client-side WebSocket URLs that need to inherit the sub-path, prefer reading `env.COLLABORATION_URL` (already path-aware via `server/env.ts:252-254`) over re-deriving the path from `env.URL` manually.
- **Build/runtime URL parity** is the operational invariant most likely to bite future contributors. Vite's `base`, the PWA manifest's `start_url`/`scope`/icons, and any URL hardcoded into hashed bundles are all baked at `yarn vite:build` time using the value of `URL` at that moment. If `URL` differs at runtime, the SPA either fails to load (manifest 404) or loads but can't fetch its own bundles (asset 404). The `.env.example` deployment-notes block documents this; reinforce it in any deployment runbook.
