# Plan: 04-subpath-realtime-and-docs

## 1. Goal

Final slice in the sub-path series. Slices 01–03 already mount the inner Koa app under `${env.BASE_PATH}`, prefix every server-rendered HTML / Vite asset URL, and wire `env.BASE_PATH` into the React Router basename, the `ApiClient` default `baseUrl`, the service-worker registration, and 65+ image / iframe literals. The two remaining surfaces are the WebSocket upgrade handlers (`/realtime` for Socket.IO presence in `server/services/websockets.ts`, `/collaboration` for Hocuspocus in `server/services/collaboration.ts`) — both register on the raw `http.Server` *before* Koa's pipeline runs, so koa-mount cannot rewrite `req.url` for them. This slice prefixes those two paths with `env.BASE_PATH` (server upgrade-handler check + Socket.IO `path` option + the `req.url?.startsWith("/realtime")` cross-service fallback in `collaboration.ts`), prefixes the matching client Socket.IO `path` option in `app/components/WebsocketProvider.tsx`, and replaces `docs/apache-vhost.conf` and the `.env.example` deployment-notes block to document the shared-subdomain layout (with WebSocket rewrite scoped to `/kms/`) as the canonical recipe — keeping the dedicated-subdomain example as an alternative section so existing deployments retain a reference. The Hocuspocus *client* (`app/scenes/Document/components/MultiplayerEditor.tsx:84`) needs no code change because `env.COLLABORATION_URL` already inherits `URL`'s path component (line 252 of `server/env.ts`) — but a test asserts the round-trip explicitly. After this slice lands, presence avatars, live cursors, and collaborative editing all work end-to-end under `/kms/`, and ops has a single deployment recipe to copy.

## 8. Risks & open questions

1. **Coupling between `websockets.ts:33` and `collaboration.ts:111`.** Both files now derive a `/realtime` path from `env.BASE_PATH`. If a future contributor changes the realtime path (e.g., to `/socket.io`), they must update both sites or the cross-service fallback will silently break. **Mitigation options:** (a) leave a one-line comment on `collaboration.ts:111` linking to `websockets.ts:33` (low cost, easy to miss); (b) export a shared constant from a new `server/services/websocketPaths.ts` (clean, but expands the slice diff by a new file and an import). **Decision:** apply (a) for this slice — minimal-diff principle. Note as follow-up cleanup in `completed.md` for option (b) if the value is referenced again later.

2. **Testing `WebsocketProvider`'s `path` option.** The component is wired through `withTranslation()` and `withStores()` HoCs and instantiates ~50 `socket.on(...)` listeners on mount. Mounting the full component in jest+jsdom is heavyweight. **Two test options:** (a) extract a helper `getRealtimePath()` exported alongside the component, unit-test the helper; (b) mock `socket.io-client`'s `io` export with `jest.fn()` and mount the component, assert against the recorded `path` arg. **Decision:** prefer (a) — a 5-line helper, a 10-line test, no HoC shenanigans. The helper is a pure function of `env.BASE_PATH`; the test exercises the same code path the component uses (`createConnection` calls `getRealtimePath()`). **Caveat:** if the test infrastructure for the `app/` jest project is still broken (slice 03 flagged `reflect-metadata` polyfill missing), the test won't run regardless. Document in `completed.md` and rely on Phase E manual smoke.

3. **Test driver shape for upgrade handlers.** `server/services/websockets.ts` and `collaboration.ts` register `'upgrade'` event handlers on the raw `http.Server`. The test driver must (a) stand up a real `http.createServer()` (not a Koa-mounted one, since the upgrade handlers attach to the raw server), (b) call `init(stubKoaApp, httpServer, serviceNames)`, (c) emit `'upgrade'` events synthetically with `httpServer.emit('upgrade', mockReq, mockSocket, Buffer.alloc(0))`. **Open question:** Socket.IO 4.x's `IO.Server` constructor attaches its own internal upgrade listener; when we invoke `init`, the listener sequence is non-trivial (the existing code in `websockets.ts:52-60` pops and re-registers it). The test driver needs to capture the popped listener via the same trick. **Mitigation:** the test should be written defensively — assert against observable side effects (`socket.end` called or not called), not against listener identity. Drives the test toward integration shape rather than fine-grained unit, which is fine for a small handful of cases.

4. **`PublicEnvironmentRegister` cache for `env.public.BASE_PATH`.** Slice 01's known issues flagged that `@Public` values are snapshotted at first `process.nextTick` after `Environment` construction and don't refresh when `env.URL` is mutated at runtime. This is a *test-only* concern: in production, `env.URL` doesn't change after server boot. In tests, the integration tests for the upgrade handlers exercise the *server-side* `env.BASE_PATH` getter (live, not the `env.public` snapshot), so the cache issue doesn't apply here. The client-side `WebsocketProvider` test reads `window.env.BASE_PATH`, which the test driver controls directly via `window.env = { BASE_PATH: ... }`. No mitigation needed in this slice.

5. **Apache rewrite-rule ordering.** The `RewriteEngine On` directive may already be set in another VHost or `.htaccess`. The new vhost should set `RewriteEngine On` explicitly inside each VHost block to avoid relying on inherited state. The plan's vhost rewrite includes this; verify during Phase D.

6. **WebSocket rewrite scoping (multi-tenant safety).** Spec §technical-notes is explicit: with multiple apps on `appstpcid` (KMS, `inventory journal plog`, `pgl`), a host-wide WebSocket rewrite rule (`RewriteRule /(.*) ws://127.0.0.1:3100/$1`) would intercept ALL WebSocket upgrades on the host and route them to KMS's port — breaking the other apps' real-time features. The new rule MUST scope to `/kms/(.*)`. **Mitigation:** the rewrite rule is the most security-/correctness-sensitive line in the new vhost; double-verify during Phase D that the source path is `/kms/(.*)` and that the destination embeds `/kms/$1` (so the prefix is preserved through the proxy hop, matching the Koa mount). Add a multi-line `#` comment above the rewrite rule explaining this constraint so a future ops engineer doesn't "simplify" the rule by removing the scope.

7. **Socket.IO `path` and the Engine.IO `?EIO=4&transport=...` query string.** Socket.IO's client appends a query string to the path during the handshake. The server-side `req.url?.startsWith(path)` check at `websockets.ts:65` works correctly because `startsWith` matches the prefix; the query string lives after the path. With the new prefixed path, `req.url` becomes `/kms/realtime/?EIO=4&...` and `path` becomes `/kms/realtime` — the `startsWith` still matches. Verify in the integration test by emitting an `'upgrade'` event with a `req.url` that includes the query string.

8. **Trailing-slash robustness in the Hocuspocus client URL.** `${env.COLLABORATION_URL}/collaboration` with `COLLABORATION_URL=ws://host/kms` resolves to `ws://host/kms/collaboration`. With `URL=http://host/kms/` (trailing slash), `server/env.ts:253` strips the trailing slash, so `COLLABORATION_URL` is still `ws://host/kms`. The slice 02 `parseBasePath` helper already normalizes this on the server side. Confirm via the round-trip assertion in `collaboration.test.ts` that both `URL=http://host/kms` and `URL=http://host/kms/` produce the same client URL.

9. **`pingTimeout`/`pingInterval` and disconnect detection.** `websockets.ts:40-41` sets `pingInterval: 15000, pingTimeout: 30000`. The smoke walkthrough's "close Browser A, Browser B's presence drops within ~30s" assertion depends on these values. If the smoke test fails to drop presence within the window, the issue is unlikely to be the path-prefix change (which doesn't affect the heartbeat). The Hocuspocus side has its own heartbeat (`debounce: 3000, timeout: 30000` — `collaboration.ts:40-41`) — confirm both timers tick during the smoke test by waiting at least 60s with a single browser open before declaring "disconnect detection works."
